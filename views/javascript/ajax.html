# ajax

# ajax 的核心

ajax 的核心就是XHLHttpRequest(XHR)；XHLHttpRequest对象用于与服务器交互。通过 XHLHttpRequest 可以在不刷新页面的情况下请求特定 URL，获取数据。

# ajax 的四部曲

  • 1. 创建 XHR
    • 不兼容 XMLHttpRequest 的浏览器(ie)使用 ActiveXObject 创建
    let xhr = new XHLHttpRequest()
  • 2. 打开 URL(发送请求前的一些处理)
    • xhr.open(method, url, async, user-name )
    • method
      • get 系列：从服务器获取
      • post 系列：向服务器发送信息
      • ...
    • async是否为异步请求，默认是true，也就是异步，设置为false代表当前请求任务为同步（项目中基本上采用异步请求）
    • user-name、user-pass向服务器发送请求所携带的用户密码，只有在设置服务器设置了安全来宾账户的情况下需要（一般不用）
    • 监听请求状态，在不同状态中做不同的事情
    • 发送 ajax 请求（ajax 任务开始，直到响应主题信息返回【ajax 状态为 4】代表当前任务结束）
  • 3. 监听状态和获取数据
    • 通过监听不同的状态来实现数据的获取
  • 4. 发送请求（请求主体的信息基于send的时候发送给服务器）

# 属性

  • xhr.response 返回响应的正文。返回的类型为 ArrayBuffer 、 Blob 、 Document 、 JavaScript Object 或 DOMString 中的一个。 这取决于 responseType 属性。
  • xhr.responseText 服务端响应的数据
  • xhr.responseType 定义响应类型值
  • xhr.status 响应的HTTP状态
  • xhr.statusText HTTP状态的说明
  • xhr.timeout 请求的超时事件
  • xhr.withCredentials 一个布尔值，用来指定跨域 Access-Control 请求是否应带有授权信息，如 cookie 或授权 header 头
  • xhr.upload 属性返回一个 XMLHttpRequestUpload 对象，用来表示上传的进度。

# HTTP 请求方法

# GET 系列

  • 特点： 一般认为是服务器获取信息（当然也可以把客户端的信息传递给服务器），比重：给的少，拿的多
  • 分类： GET、DELETE、HEAD、OPTIONS
    • get
    • head： 只获取响应头的信息即可，响应主体信息不接受（服务器不需要返回）
    • delete: 一般应用与想删除服务器上的一些文件或者一些大量信息
    • options： 试探性请求，常用于 CROSS 跨域资源共享的时候，每一个正常的请求发送前，我们默认先发一个 options 请求，这个请求用来校验客户端和服务器是否正常链接即可
  • 基于 get 向服务器发送请求，传递给服务器的方式：
    • 基于请求头传递给服务器
    • 请求 url 地址后面的问号传参（主要方式）
    xhr.open('get', './data.json?id=2&lx=0')
xhr.send()

# POST 系列

  • 特点： 一般认为是给服务器推送信息（当然服务器也可以给客户端返回信息），比重：给的多，拿的少
  • 分类： post、put
    • post
    • put： 和 delete 对应的，一般用于给服务器传递文件或者大数据（例如文本编辑器编辑的内容）
  • 基于 post 向服务器发送请求，传递服务器的方式：
    • 基于请求头传递给服务器
    • 基于请求主体，把信息传递给服务器（主要方式）
    xhr.open('post', './data.json')
xhr.send(data)

# GET 和 POST 系列的区别:

官方文档没有明确规定 GET 和 POST 的区别，但是大家约定俗称的按照以下的方式：

  • GET 传递给服务器的信息一般基于URL地址问号传参实现
  • POST 传递给服务器的信息一般基于请求主体实现
  • 客户端还可以基于设置请求头把一些简要的信息传递给服务器（例如cookie、token...）
  • GET 传递给服务器的信息要远远小于 POST
    • 原因： URL 在不同浏览器中有长度的限制 (IE: 2kb（2*1024B）)，超出浏览器限制的部分，内容会被自动裁切掉；POST 理论上没有长度限制（请求主体设置过大小的限制）,但是真实项目中为了保证数据传输的高效性，我们都会手动做限制；
  • 安全问题：POST 相对 GET 来讲安全好一些
    • 原因： 项目中涉及安全信息的传输都要用 POST（例如：登录）；GET 基于 URL 传输数据很容被 URL 劫持掉，这样不安全；POST 相对安全，但也不是绝对安全，所以对于重要的传输也需要进行手动加密处理（MD5 加密等）；
  • 缓存问题：get 请求容易产生缓存
    • 原因： 浏览器在处理 GET 请求的时候，如果两次请求的地址 + 参数都一致，浏览器自己会设置数据缓存（当然这个缓存是我们不想要的）；所以，有时候解决这个问题就会给参数后面加随机数或一个时间戳等。
